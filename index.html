Data Source

Instacart is an American company that operates as a sometimes-same-day grocery delivery service. Customers select groceries through a web application from various retailers and the order is delivered by a personal shopper. To predict which products will be in a user's next order, they released a relational dataset of files describing customers' orders over time. For more information, click here.

Data Size

The dataset is anonymized and contains a sample of over 3 million grocery orders from more than 200,000 Instacart users. For each user, we provide between 4 and 100 of their orders, with the sequence of products purchased in each order.

Data Format

The dataset consists of these different files: orders, products, aisles, deptartments, and order_products (prior, train, test). Most of the files and variable names should be self-explanatory. For more descriptions, see here.

Target

In this project, we will first do some exploratory analysis on the dataset and then develop a model to predict which previously purchased products will be in a user’s next order.

Introduction

In this part, we are going to ask a question about the data that can be solved later.

There exists a general trend that online shopping becomes more and more popular. The quality of recommendation for customers is very significant to the success of an online store or platform. Good recommendation system can provide a wonderful shopping experience, let customers get indulged in the fun of online shopping, so that platform can make more profits. For this project, we use the Instacart dataset, which contains a sample of over 3 million grocery orders from more than 200,000 Instacart users. On one hand, we do analysis of the data, include exploratory analysis, data quality analysis, statistic analysis, and visual analysis. For visual analysis, we use tableau and d3js to analyze the relationship between customers, products, orders and reorders. On the other hand, we will build an engine to predict which product will be reordered and give recommendations to users. Some kernels we are experimenting on include collaborative filtering, lightGBMand K-means clustering. We will also build a backend database server and implement the interactive recommendation on the final website.

We want to know for a user, which product in his / her previous shopping list, he will buy again. It is a practical problem since shopping platform can do recommendation base on that prediction in order to make more profit.

Accessibility

Who has access to the data?

We got the data from the official website of Instacart. (Link for the dataset : here.) They collected a relational set of files describing customers' orders over time. Concretely, dataset contains a sample of over 3 million grocery orders from more than 200,000 Instacart users. For each user, less than a hundred their orders were listed, with the sequence of products purchased in each order. Datasets also include the week and hour of day the order was placed, and a relative measure of time between orders.

Actually all the people have the access to the data for research and non-commercial usage. Instacart also want people help them to develop better recommendation system.

Completeness

Is data complete?

Yes, this data is basically complete. We can find following attributes from 7 different tables: For every order, there are user_id, order_id, order_hour, order_weekday, days_since_prior_order and id for all products in the orders provided. With product_id, we can find product_name, id and name of products’aisle, id and name of products’department. Most importantly, we get a bool attribute present whether a product in an order was reordered after that, so that we can build model to predict for a new customer, which its previous ordered product will be reordered, and then give recommendation. To verify the completeness of the data, we check following items: whether product information and order information and user_id exist for every order, whether every product has a valid name and aisle / department id, whether every aisle_id has valid aisle name and whether every department_id has valid department name. Basically, all the checks are passed, only a few unclearness about the meaning of data. For aisle name and department name, they add “other” and missing as a special name to maintain data completeness, as shown in figure 1, missing and other value contains 5.19% and 1.10% of all products.

Therefore, we just remove products with that special aisle/department name since they are not meaningful. For attribute days_since_prior_order, it the order is a user’s first order shown in the dataset, this attribute is empty. Actually these empty data has no meaning, so we just fill in them with zero.

Coherence

Does the data make sense relative to itself? Does it match our expectations? Is the data is distributed across its possible values in a way that makes sense?

The data we got has been cleaned and well formed, so it makes sense relative to itself very well. We verify the most important issue: whether a product is reordered by a user. On one hand, we can get it directly from attribute reordered. On the other hand, we compute it by grouping all the orders based on user. And then for all the orders from single user, we can find out which products are reordered, and which are not. We compare results get from two methods and find data is relative to itself. This result just matches our expectations.

The data distribution looks make sense for us. We first check reorder rate. In sampled training data, 828824 products (accumulate counted) was reordered while 555793 products are not. More than half products are reordered just in accordance to people’s shopping style. Then we check distribution of orders on different time. From figure 2, we see that people place most order on Sunday. Maybe due to people have more time on Sunday to do shopping. And more orders are placed during daytime, from 9am to 5pm. We guess that is because people have computers opened during work hours, so they have more chance to place order during then.

Exploration

Exploratory analysis of the data

The fundamental exploratory analysis is done on the products. We calculate the reorder rate according to the transaction data, and categorized products in terms of department and aisle. As is depicted in the following figure, products in the department in daily eggs are most possible ones to be bought again. This is quite understandable since people will have a periodic requirement for products like eggs, milk, butter, and they are very likely to choose the same brand. The least possible department is personal care, including skincare, first aid, etc. It’s also reasonable since these products are not so frequently needed, and people can choose different kinds of brand, or drug types.
